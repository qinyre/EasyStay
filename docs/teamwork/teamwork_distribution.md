# 易宿酒店预订平台 - 三人分工详细说明书

## 0. 协作核心原则：接口先行

为了实现**进度互不干扰**，项目启动后的前两个小时，三人必须共同确认 `docs/technical/api_spec.md` 中的字段名。一旦确认，前端 A 和 B 使用 Mock 数据（静态假数据）开发界面，后端 C 独立开发接口逻辑。

---

## 1. 成员 A：移动端前端工程师 (Mobile Developer)

**核心目标**：负责 C 端用户体验，确保预定流程流畅。

### 酒店查询页 (首页)

* 实现顶部广告 Banner 及其跳转逻辑
* 实现搜索栏、定位功能及快捷标签展示
* **技术难点**：独立开发**日历组件**，支持入住与离店日期选择

### 酒店列表页

* 实现多维度筛选头（城市、日期、星级等）
* **技术得分点**：开发**长列表渲染优化**，支持上滑自动加载新数据

### 酒店详情页

* 实现图片轮播 Banner
* **逻辑难点**：对后端返回的房型数据进行**价格由低到高排序**呈现

---

## 2. 成员 B：PC 管理端前端工程师 (PC Admin Developer)

**核心目标**：负责 B 端管理逻辑，确保商户与管理员操作闭环。

### 登录与注册页

* 开发注册界面，支持"商户"与"管理员"角色选择
* 实现登录逻辑，需根据后端返回的账号信息**自动判断并跳转**对应后台

### 酒店信息录入/编辑页 (商户端)

* 构建复杂表单，涵盖酒店名（中/英）、星级、价格、开业时间等**必须维度**
* 实现数据保存功能，并确保实时触发同步

### 审核与下线列表页 (管理员端)

* 开发审核功能，支持填写"不通过原因"
* 实现**"下线/恢复"切换逻辑**（即虚拟删除功能）

---

## 3. 成员 C：后端全栈工程师 (Backend & DB Coordinator)

**核心目标**：负责数据持久化、接口实现及全局文档。

### 服务端架构

* 使用 Node.js 搭建基础服务框架
* 设计并维护 `hotels.json` 和 `users.json` 的存储结构

### API 接口实现

* 编写用户认证接口（注册、登录、角色判定）
* 编写酒店 CRUD 接口（增删改查），特别是审核状态的变更逻辑

### 数据一致性保障

* 实现文件读写锁逻辑，确保 JSON 数据库在高频保存时不损坏
* **技术得分点**：确保商户端修改后的数据能**实时通过 API 更新**至移动端

### 公共交付物

* 负责编写最终的 `README.md` 项目运行指南

---

## 4. 进度同步路线图

1. **第 1 天**：三人对齐 API 字段（docs 目录搭建完成）
2. **第 2-5 天**：A 攻克日历与长列表；B 攻克录入表单；C 完成 JSON 读写接口
3. **第 6 天**：全面联调，将前端地址由本地 Mock 切换为 C 的后端地址
4. **第 7 天**：对照测试文档进行最终验收
